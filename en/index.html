2026
Gmarket

Pricing management admin — official launch and operation

Officially launched the pricing back-office built in 2025. Roughly 150 sales HQ staff now use it daily, viewing price statistics by product and channel and ranking price competitiveness against competitors in a single screen.

Large CSV downloads combine StreamingResponseBody with MongoDB cursor streaming, writing row by row without loading the full dataset into memory — server memory stays flat even for hundreds of thousands of rows. UTF-8 BOM injection handles Korean Excel compatibility, and field projection trims query load.

The front-end is framework-free ES Modules, split into page modules (summary/items) and shared modules, with a snapshot-based state module managing filters, selections, and panels. Fetch ReadableStream reads received chunk sizes to drive a download progress status bar with a current-filter vs. full-export scope menu. Per-user column layouts and page sizes persist in localStorage, with merge logic so new columns don't break existing arrangements.

The backend separates operational reads/writes (MongoDB) from pipeline output queries (Databricks JDBC) as dual data sources, manages credentials via Vault, and aggregates MTD trends server-side in day/hour buckets. Domain policies and decision records are documented so AI agents can pick up work on the codebase.

Proposed and led a team domain knowledge base that humans and AI read and fill together. Currently ~330 wiki nodes and ~140 raw documents.

Built an MCP server with the standard library only: teammates just open the repo folder and their agent auto-connects, searching and citing the knowledge base — no API key required. A separate dashboard app provides a knowledge graph, ERD, workflow Gantt, and a RAG chatbot (Claude CLI integration with keyword-search fallback).

Transition from Batch to Stream for product price data synchronization

Improved the product pricing DB sync process for dynamic pricing. The original approach replicated a DB then read 400 GB of data via batch.

The batch first truncated then refilled pricing data, causing gaps that broke dynamic pricing jobs running in parallel. Resolved by staging data in a temp table and swapping with RENAME instead of truncating.

As product count grew, sync jobs began overlapping. Final solution: initial load via mysqldump, then a Kafka consumer app to apply only incremental changes in real time.

Expanded discount-eligible product coverage from 50M to 280M; achieved real-time pricing. Strengthened stability with Poison Pill handling and Row Lock-free transaction design.
